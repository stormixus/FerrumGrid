> 개정 1회: 독립 적대적 리뷰(5축, 코드/egui 소스 대조) 반영. 핵심 변경 — (a) egui 기본 입력 채널 활용으로 인터랙션 간소화, (b) 드래그/팬 판별 순서 명시, (c) Phase 0(진짜 최소 수정) 분리, (d) 카메라 리셋 위험 처리, (e) 물리 루프 상수 구체화, (f) LayoutEngine trait는 YAGNI로 후순위.

## Problem Statement

현재 ERD(Model) 뷰(`src/ui/er_diagram.rs`, 1289줄)는 세 가지 불편이 있다:

1. **보기 불편** — 테이블 표현이 큰 카드(폭 300px, 컬럼 8행) 한 종류뿐. 테이블이 많아지면 명함첩처럼 격자로 깔려 전체 구조가 한눈에 안 들어온다. 줌 범위도 `0.5~1.8`([er_diagram.rs:1090](src/ui/er_diagram.rs:1090))로 좁아 큰 스키마를 멀리서 조망 불가.
2. **드래그 불편** — 빈 캔버스 드래그=패닝, 카드 드래그=이동이 한 레이어에 섞임. 판별이 1프레임 지연 해킹(`is_panning`은 카드가 안 끌릴 때만 set, [er_diagram.rs:725-727](src/ui/er_diagram.rs:725))이라 미끄러짐.
3. **스크롤=줌이 싫음** — 마우스 휠이 `smooth_scroll_delta.y`로 줌에 직결([er_diagram.rs:1086-1091](src/ui/er_diagram.rs:1086)). 업계 표준(스크롤=팬, ⌘/핀치=줌)과 어긋남.

원하는 방향(4번째, 명시적 목표): **"획기적인 것"** — 트랜지션 애니메이션으로 살아 움직이는 그래프, Obsidian식 점(노드) 표현, 노드 형태의 다양화. (정직하게: 작업의 ~90%는 1~3 불만이 아니라 이 wow 목표를 위한 것. 1~3만이면 Phase 0의 <100 LOC로 끝난다.)

## What Makes This Cool

**지도식 시맨틱 줌을 DB 스키마에 적용한다.** 줌 레벨이 곧 표현을 결정:

- **멀리(줌 작음)** → Obsidian식 **점**. FK 차수(degree)로 크기, 라이브 포스 물리로 군집이 살아 움직임.
- **중간** → **컴팩트 칩**(테이블명 + PK/FK 배지 + 컬럼 수).
- **가까이(줌 큼)** → 현재 **풀 카드**(컬럼/타입/PK/FK).

전환은 부드러운 크로스페이드 모핑. 줌하는 동안 노드가 "자라나고" FK 선이 따라 흐른다.

레퍼런스: **Azimutt**(오픈소스 ERD 탐색기, 시맨틱 줌은 없음), **Obsidian 그래프**(degree 점 크기 + 라이브 포스), **dbdiagram.io/DrawSQL**(예쁘지만 정적), **Figma/tldraw/구글맵**(시맨틱 줌 + 관성 + 미니맵 표준). **차별점:** 지도식 시맨틱 줌 + 라이브 물리를 합치는 DB 클라이언트는 거의 없다.

## Constraints

- **egui 즉시 모드** — 매 프레임 전체 재그림. 물리/애니/카메라가 움직이는 동안 `ctx.request_repaint()`, 안정 시 멈춰 idle CPU 0.
- **재사용 가능 자산(검증됨)** — force-directed 레이아웃([er_diagram.rs:130](src/ui/er_diagram.rs:130)), `world_to_screen`([er_diagram.rs:1128](src/ui/er_diagram.rs:1128)), 베지어 FK([er_diagram.rs:255](src/ui/er_diagram.rs:255)), 테마/배지/검색 모두 존재. **단, force는 "200회 블로킹 루프 + 끝에서 bbox 재중심"이라 라이브化는 재사용이 아니라 함수 본문 재작성에 가깝다.**
- **기존 기능 보존** — 더블클릭→데이터 열기([er_diagram.rs:876](src/ui/er_diagram.rs:876)), FK 베지어, PK/FK 배지, 검색([er_diagram.rs:370](src/ui/er_diagram.rs:370)), 스키마 콤보, Tab/Esc 내비.
- **데이터 비동기 로딩 위험(중요)** — `sync_schema_visualizer`는 카드 키셋이 바뀔 때마다 `layout_cards()` + `pan_offset = ZERO`를 호출([er_diagram.rs:1245-1248](src/ui/er_diagram.rs:1245)). 컬럼은 테이블별 lazy 로드. 라이브 물리/제스처 중 테이블이 늦게 도착하면 **카메라가 원점으로 튀어** 사용자 팬/줌과 싸운다. → Phase 1 진입 전 반드시 수정.
- **테마** — Dot/Chip 렌더러도 `theme::is_dark()` 분기를 카드와 동일하게 적용(그림자 알파 등).

## Premises (확정됨)

1. 큰 스키마(수십~수백 테이블)도 핵심 타깃. ✅
2. 기존 카드 기능 유지. ✅
3. 60fps 부드러움 + settle 시 idle 0. ✅
4. 단계적 출시(Phase 0 → 1 → 2 → 3). ✅

## Approaches Considered

### Approach A: 현 캔버스 최소 증축 (minimal viable)
LOD enum을 zoom에서 파생, draw_card를 LOD 분기 + 크로스페이드로 확장. 스크롤=팬/⌘·핀치=줌, 줌 0.15~2.5. 포스 라이브. Effort S~M / Risk Low. 한 파일이 비대해지고 새 폼 추가 시 if 분기 증가.

### Approach B: 점진적 이상형 (ideal architecture) — 채택
`GraphScene`(노드=현재pos+목표pos+속도) + 보간 애니메이터 + LOD 렌더러 + (후순위) 레이아웃 추상화. **Phase 0/1 = Approach A 산출물**로 바로 출시, 이후 모듈화/엔진화. Effort L(단계 분할) / Risk Med. 전환이 한 애니메이터로 일관, 형태 다양화 확장 용이.

### Approach C: 그래프 크레이트 도입 (creative/lateral)
`egui_graphs`/`fdg` 위임. Effort M / Risk Med-High. 코드량↓이나 위젯 노드 렌더가 카드 미감·테마·시맨틱줌 모핑과 충돌, 기존 베지어/배지/더블클릭을 위젯 안에서 재현 필요.

## Recommended Approach

**Approach B (점진적 이상형), Phase 0를 먼저 단독 출시.** 직접 구현을 크레이트보다 우선하는 이유: 카드 미감 + 시맨틱 줌 모핑이 제품 핵심인데 범용 위젯은 이와 싸우고, force·베지어·테마가 이미 있어 크레이트로 가면 잃는다.

### 인터랙션 모델 (egui 기본 채널 활용 — 리뷰 핵심 반영)

egui가 이미 처리하는 것을 재발명하지 않는다:
- **줌 = `i.zoom_delta()`** — ⌘/Ctrl+스크롤 **그리고** 트랙패드 핀치를 모두 포함. 모디파이어 직접 감지 불필요.
- **팬 = `i.smooth_scroll_delta`** — egui가 줌 모디파이어가 눌린 스크롤은 이 채널에서 **이미 제외**한다. Shift+스크롤 가로 변환도 egui가 처리. 따라서 `pan += smooth_scroll_delta`만으로 스크롤=팬 + Shift=가로팬 완성.
- ⚠️ 현재 코드는 `smooth_scroll_delta.y`로 줌을 읽음([er_diagram.rs:1086](src/ui/er_diagram.rs:1086)). 새 코드에서 팬을 이 채널로 읽으면서 줌도 따로 모디파이어로 읽으면 **이중 처리**가 된다. → 줌은 `zoom_delta()` 단일 소스로.

**zoom-to-cursor 공식** (커서 밑 월드점 고정, `world_to_screen = canvas.min + pan + world*zoom`의 역):
```
let c = pointer_pos;                  // 화면 좌표
let z1 = (zoom * zoom_delta).clamp(0.15, 2.5);
pan = pan + (c - canvas.min - pan) * (1.0 - z1 / zoom);
zoom = z1;
```

**드래그 vs 팬 판별 (순서 반전 — 리뷰 #1):** 현재의 1프레임 지연 `is_panning`을 버린다. 한 프레임 안에서:
1. 모든 노드를 z-order(위→아래)로 `ui.interact()` (각자 고유 `Id`).
2. 어떤 노드가 `drag_started()/dragged()`면 `node_consumed_drag = true` + 그 노드를 드래그(이동+pin).
3. `!node_consumed_drag && bg_response.dragged()`일 때만 팬.

점은 히트 타깃이 작으니 이 순서가 필수(안 그러면 팬 시작했다가 다음 프레임 점이 가로채 오발).

### 시맨틱 LOD + 크로스페이드
- `zoom < 0.45` → **Dot** (반지름 = f(degree), 라벨은 hover/선택/검색매치 시만 — Obsidian식)
- `0.45 ≤ zoom < 0.9` → **Chip**
- `zoom ≥ 0.9` → **Card**
- 경계 ±0.08 밴드에서 **두 표현을 모두 그리되** alpha = 선형 램프(0.16 줌 단위에 걸쳐 fade). egui는 raw `Painter::rect_filled`에 공짜 투명도가 없으므로 둘 중 하나:
  - (권장) 해당 LOD 렌더를 별도 레이어로 그리고 `ui.set_opacity(a)` 적용, 또는
  - 모든 paint 색상에 알파 곱(`theme::with_alpha`)을 통과시키는 `opacity: f32` 인자. `draw_card`(~200줄) 리팩터 필요.

### 라이브 물리 루프 (상수 구체화 — 리뷰 #3)
- 기존 baseline 재사용: `damping 0.85`, `spring_k 0.05`, `repulsion 5000`([er_diagram.rs:144-148](src/ui/er_diagram.rs:144)).
- 프레임당 **3 iteration**. **매 프레임 bbox 재중심 금지**(드리프트로 사용자 팬과 싸움) — 최초 1회만 중심.
- **settle:** 총 운동에너지 `Σ|v|² < 0.5`(튜닝)인 프레임이 **연속 30회**면 정지 + repaint 중단.
- **pin:** 드래그된 노드는 속도 적분 제외, pos = 커서 월드좌표. 드래그/데이터변경 시 settle 카운터 리셋(re-heat).
- `노드 < 2`면 force 스킵(degenerate bbox), grid로.
- **성능:** all-pairs 반발은 O(N²). 라이브 물리는 **N ≤ 80**까지만 매 프레임; 초과 시 1회 settle 후 정지(애니만 유지), Barnes-Hut/그리드 버킷은 Phase 3 과제로 명시. "100+ 테이블 프레임드랍 없이"는 따라서 **Phase 2+ 목표**(아래 성공 기준 단계 태그 참조).

### 카메라 / 애니메이터
- `Camera { pan, zoom, target_pan, target_zoom }` — 줌/팬을 target으로 ease(관성).
- **repaint 조건 = `physics_hot || any_node_pos != target || camera != target`** (애니/카메라도 repaint 구동, 물리 settle만으론 부족 — 리뷰 #2).
- 레이아웃 전환/정렬/fit = `target` 설정 → ~300ms ease-out 보간.

### 뷰포트 컬링 (리뷰 #4)
- 노드/FK 모두 `canvas_rect`(+마진) 밖 월드좌표는 그리기·interact 스킵. 현재 FK 컬링은 검색 필터(`visible_ids`)만 봄([er_diagram.rs:920](src/ui/er_diagram.rs:920)) — 뷰포트 추가.

### 모듈/추상화 (YAGNI 반영 — 리뷰 scope #2)
- Phase 2에서 `er_diagram/` 디렉터리 분리(scene/camera/animator/render/edges/interaction).
- **`LayoutEngine` trait는 두 번째 라이브 레이아웃이 실제로 생길 때까지 보류.** Phase 0~2는 enum + 함수로 충분(Grid/Force 2종). 조기 추상화 금지.

## Success Criteria (단계 태그)

- **[P0]** 마우스 휠=팬, ⌘/핀치=줌(스크롤=줌 제거). 줌 범위 0.15~2.5.
- **[P0]** 노드 드래그가 부드럽고 팬과 오발 없이 분리.
- **[P1]** 줌하면 점→칩→카드가 끊김 없이 모핑.
- **[P1]** 노드 드래그 시 이웃이 물리 반응, 손 떼면 정착. settle 후 idle CPU ≈ 0.
- **[P1]** 레이아웃 전환/정렬이 순간이동이 아닌 애니메이션.
- **[P2+]** 100+ 테이블에서 줌아웃 시 점-그래프로 한눈에, 프레임드랍 없이(컬링/대규모 물리 전략 적용 후).
- **[전 단계]** 기존 기능(더블클릭/FK/배지/검색) 전부 동작, 다크·라이트 테마 정상.

## Reviewer Concerns (잔여 — 구현 시 유의)

- 크로스페이드 opacity 경로는 egui 0.31의 `ui.set_opacity`/레이어 vs 색상 알파 곱 중 PoC로 확정 필요(둘 다 가능, 비용 다름).

## What I noticed about how you think

- "막 트랜지션 되면서 움직이던데"라고 본 걸 콕 집은 것 — 정적 다이어그램과 살아있는 캔버스의 차이를 정확히 감지. 그게 설계 중심축이 됐다.
- "옵시디언처럼 점으로", "형태의 다양화" — 한 정답이 아니라 *여러 표현*을 원한다는 신호. 그래서 단일 모드 교체가 아니라 멀티모드로 방향이 잡혔다.
- 불만을 셋(보기/드래그/스크롤)으로 분리해 말한 것 — 막연한 "별로야"가 아니라 인터랙션 단위로 쪼갠 사고. 덕분에 Phase 0가 바로 실행 가능한 <100 LOC 항목으로 떨어졌다.
